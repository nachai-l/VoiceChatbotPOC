## Bugs Fixed

### 1 — Mic stream killed after first audio response
**Symptom:** `stream_age_s` climbed to ~27 s exactly when playback started; system hung on "Listening..." with no VAD activity.

**Root cause:** `gr.Audio(autoplay=True)` — when Gradio updated this component with new audio bytes, the browser reset its AudioContext.  The echo-cancellation subsystem interpreted the reset as a new recording context and stopped the active mic MediaStream.

**Fix (`app/ui/gradio_app.py`):**
- Replaced `gr.Audio(autoplay=True, type="numpy")` output component with `gr.HTML(value="")`.
- Added `_build_audio_html()`: encodes PCM16 → WAV → base64 data URI, returns an `<audio autoplay controls>` HTML string.  `gr.HTML` updates are plain DOM text swaps with no AudioContext involvement.

---

### 2 — No audio output on 2nd turn
**Symptom:** 2nd turn task ran and completed (logs confirmed `audio=220k bytes`), but browser played nothing.

**Root cause:** `<audio autoplay>` with a `<source>` child element has weaker browser autoplay support than `src=` directly on the tag; subsequent turns don't always receive autoplay permission.

**Fix (`_build_audio_html`):**
- Changed to `<audio autoplay controls src="data:audio/wav;base64,…">` (removed `<source>` child).
- Added `controls` attribute so the user can manually click play if the browser's autoplay policy blocks subsequent turns.

---

### 3 — Chatbot conversation box always empty (Gradio 6 breaking change)
**Symptom:** Server logs showed `_consume: returning chatbot history with 2 message(s)` — data was returned correctly — but `gr.Chatbot` remained blank after every turn.

**Root cause (Gradio 6 breaking change):** In Gradio 4/5, returning `gr.update()` for a `gr.Chatbot` output was a true no-op (preserved the current value).  In Gradio 6, `gr.update()` (`{"__type__": "update"}`) is forwarded to the client, which resets the component to its default empty state.  Every 0.25 s tick from either the stream or poll handler was clearing the chatbot immediately after `_consume_finished_task` set it.

**Investigation path:**
- Verified format: `chatbot.postprocess([{"role":"user","content":"…"}])` succeeds in Gradio 6.
- Traced `postprocess_data` source — bare `gr.update()` calls `postprocess_update_dict` which sends `{"__type__":"update"}` to the frontend without a value.  Client resets component.

**Fix (`stream_audio_chunk`, `poll_pending_result`):**
- Replaced every `gr.update()` chatbot output with `transcript_handler.get_history()` in all handler paths.
- `gr.update()` is still used exclusively for `gr.State` outputs (`transcript_state`, `vad_state`, `session_state`, `summary_state`) where suppressing write-back is needed to prevent concurrent-handler races.

---

### 4 — Stuck "Playing response..." status (State race regression)
**Symptom:** After a turn completed, status showed "Playing response..." permanently and never transitioned to "Listening...".

**Root cause — two bugs introduced by the prior State race fix:**

**Bug A — Stale-copy guard falling through:**
When `_is_task_consumed` returned True in `stream_audio_chunk`, the code cleared `pending_task` locally and fell through to the listening/VAD paths.  Those paths then wrote `vad_state` with `ignore_until=0` back to Gradio State — erasing poll's just-committed `ignore_until=T+5`.  The cooldown was reset to zero every stream tick, so "Playing response..." never persisted.

**Bug B — `last_stream_at` going stale:**
Returning `gr.update()` for `vad_state` during the playing (cooldown) path meant `last_stream_at` was never refreshed in Gradio State for the full API latency + cooldown window (~8s + 5s = 13s).  After cooldown expired, `stream_age_s ≈ 13s > 2.0` triggered the `_stream_dead` false-positive, showing "Mic stream stopped".

**Fix:**
- Stale-copy guard now returns `gr.update()` for `vad_state` immediately (same as the "thinking" path) instead of clearing pending_task and falling through.  Poll's `ignore_until` is preserved.
- Playing path now returns the actual `vad_state` (not `gr.update()`).  `pending_task` is already `None` at this point so `ignore_until` cannot be reset.  Returning actual `vad_state` keeps `last_stream_at` fresh and eliminates the false-positive.

---

### 5 — Chatbot transcript lost between turns (Gradio State write-back race)
**Symptom:** Chatbot showed messages after turn 1 but was blank after turn 2.

**Root cause:** Stream's "playing" (cooldown) path wrote `transcript_handler` back to Gradio State.  If stream received its `transcript_handler` copy from Gradio State *before* poll committed the turn-1 update, stream would write the old empty handler back — overwriting the messages poll just added.

**Fix:** Stream's "playing" path now returns `gr.update()` for `transcript_state` (not the actual handler), so poll's committed transcript is never overwritten by a stale copy.

---

### 6 — Debug panel flicker (8 DOM updates/second)
**Symptom:** Debug JSON panel flickered visibly.

**Root cause:** Both `stream_audio_chunk` (every 0.25 s) and `poll_pending_result` (every 0.25 s) were each pushing to `debug_panel`, producing 8 DOM writes per second.

**Fix:** `stream_audio_chunk` now calls `_append_trace()` to accumulate entries in `vad_state.trace_log` but returns `gr.update()` for `debug_panel`.  Poll displays the accumulated trace every 0.25 s — effectively halving debug panel updates to 4/s.

## Architecture Notes

### Gradio 6 component update semantics
| Return value | `gr.State` | `gr.Chatbot` / display components |
|---|---|---|
| `gr.update()` | True no-op ✓ | Resets to default (empty) ✗ |
| Actual value | Writes to State ✓ | Updates displayed value ✓ |

**Rule going forward:** Never return `gr.update()` for `gr.Chatbot` or other display components in Gradio 6.  Use `gr.update()` only for `gr.State` outputs where suppressing write-back is intentional.

### Handler ownership model
| Handler | Owns (writes back) | Does NOT own (returns `gr.update()`) |
|---|---|---|
| `stream_audio_chunk` — thinking | Nothing | All States |
| `stream_audio_chunk` — playing | `vad_state` (last_stream_at) | `transcript_state`, `session_state`, `summary_state` |
| `stream_audio_chunk` — listening/VAD | All States | — |
| `poll_pending_result` — any path | All States | — |
| `_consume_finished_task` | All States | — |
